# Bug Report Template

| Field | Description |
|---|---|
| **Bug ID** | Unique identifier (e.g., BUG-001) |
| **Title** | Short, specific summary of the defect |
| **Reported by** | Tester name |
| **Date** | When the bug was found |
| **Environment** | OS, browser/device, build version, URL |
| **Module** | Feature area affected |
| **Severity** | Critical / High / Medium / Low (impact on the system) |
| **Priority** | High / Medium / Low (urgency to fix) |
| **Preconditions** | State required before reproducing |
| **Steps to Reproduce** | Numbered, repeatable actions |
| **Expected Result** | What should happen |
| **Actual Result** | What actually happens (the defect) |
| **Evidence** | Screenshot, recording, or log reference |
| **Status** | New / Open / In Progress / Resolved / Closed |

---

## Reporting Tips

- Write the title so a developer understands the issue without opening the report.
- Keep reproduction steps minimal — only what is needed to trigger the bug.
- Always separate **expected** from **actual** — that contrast *is* the defect.
- Attach evidence; a screenshot or log saves significant back-and-forth.
